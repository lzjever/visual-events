# Visual Events v0.3 视觉身份与场景记忆开发计划

日期：2026-06-29

当前 0.3/0.4 GA handoff 以 `docs/memory-teaching-ga-development-plan.md` 为 source of truth；本文中低层 target、region/object 相关描述不得作为当前 GA 承诺。

## 1. 版本定位

v0.3 的目标是交付“可教学的人和场景记忆 MVP”：

- 用户可以明确告诉 agent 当前画面里的某个人是谁、有什么身份信息或特点。
- 用户可以明确告诉 agent 当前整图画面是什么场景、应该如何解释。
- 服务端保存这些教学结果；以后再次看到相似的人或场景时，输出低频视觉事实和简短 `memory_context`。
- CLI 继续只负责 DDS、gaze target 和 Botified notification gate，不做身份识别、场景识别或记忆判断。
- agent 继续决定是否回应、怎么回应、是否采取后续动作。

一句话：v0.3 让机器人“被教会后能认出来”，不做自动熟人积累和复杂手势指向。

更新说明：受限的“人指向另一个人并介绍身份”已被纳入 stable memory/teaching GA 计划，作为 `target.intent=third_person_introduction` 的当前必备能力。本文中“不做复杂手势指向”仍指手指级、物体、区域和泛化手势 grounding。

当前状态说明：

- fake embedding backend、SQLite store、memory HTTP API 和 CLI Botified gate 已经可以用确定性测试覆盖。
- `tools/run_memory_e2e.py` 是手工 memory E2E 工具，默认 fake-backed，用来证明 memory 侧链、事件投影、CLI gate、v0.3 checks 和已有 v0.4 memory checks 的闭环；它不是默认发布 gate，也不是 local 模型效果证据。
- 只有显式传 `--embedding-backend local --person-model-path ... --scene-model-path ...`，且报告满足 `ok=true`、`uses_real_model_backend=true`、v0.3 `known_person_present` + `scene_activated` assertions 通过时，才能把 local backend 视为当前 PC 本地可用。

## 2. 产品边界

### 2.1 本版本必须支持

| 能力 | 行为 |
| --- | --- |
| `teach_person` | agent 指定当前目标人，服务端保存 person profile、视觉特征和用户给的身份/特点文本 |
| `known_person_present` | 以后看到已教学的人，服务端低频输出已知人物事件 |
| `teach_scene` | agent 指定当前整图，服务端保存场景说明和视觉特征 |
| `scene_activated` | 以后看到相似场景，服务端低频输出场景激活事件 |
| `memory_context` | Botified 通知 agent 时，附带匹配到的人物/场景背景 |
| `attach_conversation_summary` | agent 把某人的短对话摘要挂到 `person_id`，后续识别到该人时返回 |
| `link_external_user_ref` | agent 显式绑定外部消息用户 ID 和视觉人物 ID，用于“这个人发消息时也能拿到元信息” |

v0.3 的目标指定方式只做一种主线：server 的 target resolver 支持 `attention_target`、`track_id`、`bbox`、`point_uv`、`scene`。但 `teach_scene` v0.3 只开放 `target.mode=scene` 的整图教学；bbox/point/region 形式的 `teach_scene` 必须明确拒绝且不写入，等查询侧支持同类 region query 后再开放。不要同时实现 CLI 命令、WebSocket command 和多套目标选择逻辑。

### 2.2 本版本不做

- 不做复杂/泛化手势指人识别作为主能力；受限的 person-to-person introduction 已由 stable memory/teaching GA 计划覆盖。
- 不做“经常出现但未命名的人”自动熟人聚类。
- 不做复杂身份纠错和匿名身份合并。
- 不自动从所有聊天原文中提取长期记忆。
- 不建设管理后台、审核后台、release audit 或治理平台。
- 不引入独立向量数据库服务。
- 不做多店铺、多租户、云同步。
- 不要求 RK3588 运行视觉记忆模型；当前验收仍是 PC 本地模拟。
- 不让 memory retrieval 决定机器人是否说话、说什么或如何运动。

内部私有场景下，隐私合规不是本版本的产品阻塞点。v0.3 延后复杂身份能力的原因是误识别、错绑人、实时延迟和实现规模风险，而不是合规治理。

## 3. 总体架构

实时主链路保持不变：

```text
DDS JPEG
  -> visual-events-cli
  -> WebSocket /v1/stream
  -> visual-events-server inference / tracking / events / attention
  -> visual_state @10Hz
  -> visual-events-cli
      -> DDS gaze target
      -> low-frequency Botified request
```

v0.3 新增低频 memory 侧链：

```text
server recent frame cache
  -> target resolver
  -> memory embedding backend
  -> SQLite memory store
  -> low-frequency memory retriever
  -> semantic_events + memory_context
  -> CLI Botified notification gate
```

硬边界：

- memory 侧链不能阻塞 10Hz inference / tracking / attention / gaze。
- memory query 只在稳定目标、教学请求、场景变化或 agent 主动请求时低频执行，不能每帧同步执行。
- memory 失败时，v0.2 已有 detection、tracking、event、gaze 和 Botified 输出必须继续工作。
- server 输出视觉事实和记忆上下文；agent 才决定业务动作。

Memory event 回到 CLI 的唯一做法：

- `/v1/stream` 仍是一帧请求对应一帧 `visual_state` 响应，不改成 server push。
- `MemoryService` 是 app-level 单例，维护 per-camera recent frame cache、pending query 和 completed memory event queue。
- 每次 `process_frame()` 完成 inference / tracking / attention / v0.2 events 后，把 frame、compact visual_state 和当前 track snapshot 写入 app-level cache，再从同一 camera 的 completed memory event queue 取出已完成结果，追加到本帧 `semantic_events`。
- 如果 embedding worker 还没完成，本帧不等待；结果在后续新鲜帧返回。过期结果直接丢弃，不补发旧帧事件。
- teaching HTTP API 也只从同一个 app-level cache 取目标；无活跃 stream、帧过期或目标属于旧连接时返回明确错误，不写入记忆。

## 4. 代码结构

新增 server 内部模块：

```text
src/visual_events_server/memory/
  __init__.py
  frame_cache.py
  target_resolver.py
  embedding.py
  store.py
  retriever.py
  events.py
  service.py
```

职责：

| 模块 | 职责 |
| --- | --- |
| `frame_cache.py` | 保存最近 N 秒 JPEG frame、frame metadata 和 compact visual_state 摘要 |
| `target_resolver.py` | 把 `attention_target`、`track_id`、`bbox`、`point_uv`、`scene` 解析成图像区域 |
| `embedding.py` | 定义 `MemoryEmbeddingBackend`，提供 fake / disabled / local backend |
| `store.py` | SQLite metadata 和 normalized vector BLOB 存取 |
| `retriever.py` | 封装 `sqlite-vec` 向量查询、threshold、top1/top2 margin 和 top_k 限制 |
| `events.py` | 生成 `known_person_present`、`scene_activated`，做 cooldown |
| `service.py` | app-level memory service，串接 teach/query/link/summary API、per-camera cache、completed event queue 和低频 memory worker |

不新增独立服务进程，不新增第二套 streaming 协议。

## 5. 服务端接口

v0.3 只新增一组 HTTP API。不要把教学命令塞进 `/v1/stream` WebSocket，也不要让 CLI 另造一套 IPC。

所有 HTTP API 都通过 app-level `MemoryService` 访问最近帧缓存。请求中的 `camera` 必须能匹配到新鲜 stream；target 解析必须带上 `connection_id`、`frame_id` 和 `frame_timestamp_ms`，避免使用旧连接或过期 track。

### 5.1 教学人物

`POST /v1/memory/teach/person`

```json
{
  "request_id": "teach-001",
  "camera": "front",
  "target": {
    "mode": "attention_target"
  },
  "profile": {
    "display_name": "张三",
    "description": "店长，熟悉新品陈列和现场活动",
    "tags": ["staff", "manager"]
  }
}
```

目标也可以使用：

```json
{"mode": "track_id", "track_id": 7}
{"mode": "bbox", "bbox_xyxy": [320, 120, 520, 600]}
{"mode": "point_uv", "point_uv": [420, 360]}
```

响应：

```json
{
  "ok": true,
  "person_id": "person_000001",
  "embedding_count": 1,
  "target_quality": "usable"
}
```

规则：

- `track_id` 找不到、帧过期、目标区域太小或 embedding 质量不足时，返回明确错误，不写入错误记忆。
- v0.3 的已知人物身份识别只走一条正式路径：从 resolved person crop 中检测人脸，再提取 face embedding。没有可用人脸时，`teach_person` 返回 `no_usable_face`，不写入 `known_person_present` 可用的身份 profile。
- person crop appearance 可以作为后续候选能力，但 v0.3 默认不把全身外观用于长期人物身份识别，避免换衣服、遮挡和相似外观导致错认。
- `local` backend 的 `person_model_path` 指向本地 face detector + face embedding 模型或模型 bundle；必须显式配置。server 不隐式下载模型，不写系统目录或用户目录。

### 5.2 教学场景

`POST /v1/memory/teach/scene`

```json
{
  "request_id": "scene-001",
  "camera": "front",
  "target": {
    "mode": "scene"
  },
  "memory": {
    "title": "新品展示场景",
    "description": "这是本周新品展示场景，主推夏季轻量外套。",
    "activation_hint": "看到顾客停留在这个场景中时，可以介绍新品活动。"
  }
}
```

v0.3 不支持 bbox/point/region 形式的 `teach_scene`。以下 payload 必须返回明确错误且不写入 scene memory，直到 `scene_activated` 支持同类 region query：

```json
{"mode": "bbox", "bbox_xyxy": [120, 90, 540, 620]}
{"mode": "point_uv", "point_uv": [310, 300]}
```

### 5.3 对话摘要

`POST /v1/memory/person/{person_id}/conversation-summary`

```json
{
  "summary": "上次问过新品尺码，偏好浅色外套。",
  "source": "agent",
  "source_conversation_id": "conv-abc"
}
```

规则：

- server 保存 agent 提供的短摘要，不保存大段原始聊天记录。
- server 不负责从聊天原文中抽取记忆。
- 返回给 agent 的摘要必须短，默认只取最近少量条目。

### 5.4 外部消息用户绑定

`POST /v1/memory/link-external-user`

```json
{
  "person_id": "person_000001",
  "external_user_ref": "wechat:staff-zhangsan"
}
```

`GET /v1/memory/person/by-external-user/{external_user_ref}`

用途：

- 当某个外部消息用户向 agent 发消息时，agent 可以查询这个用户是否已经绑定视觉人物。
- 视觉服务不能凭一条文字消息自动猜“发消息的人就是镜头里这个人”；必须依赖显式绑定。

## 6. 数据存储

v0.3 使用 SQLite + `sqlite-vec`。不要在本阶段引入 FAISS、Qdrant、Postgres 或独立 DB 服务。

最小表：

| 表 | 字段 |
| --- | --- |
| `person_profiles` | `person_id`、`display_name`、`description`、`tags_json`、`created_at_ms`、`updated_at_ms` |
| `person_embeddings` | `embedding_id`、`person_id`、`embedding_type`、`embedding_model`、`embedding_version`、`embedding_dim`、`source_target_type`、`vector_blob`、`quality`、`created_at_ms` |
| `scene_memories` | `scene_id`、`title`、`description`、`activation_hint`、`target_type`、`created_at_ms`、`updated_at_ms` |
| `scene_embeddings` | `embedding_id`、`scene_id`、`embedding_type`、`embedding_model`、`embedding_version`、`embedding_dim`、`source_target_type`、`vector_blob`、`quality`、`created_at_ms` |
| `conversation_summaries` | `summary_id`、`person_id`、`summary`、`source`、`source_conversation_id`、`created_at_ms` |
| `external_user_links` | `external_user_ref`、`person_id`、`created_at_ms` |
| `memory_match_records` | `memory_match_id`、`event_id`、`matched_type`、`matched_id`、`embedding_id`、`match_score`、`top2_margin`、`source_target_mode`、`camera`、`frame_id`、`frame_timestamp_ms`、`created_at_ms` |

向量规则：

- embedding 统一保存为 normalized float32 BLOB，并写入 `sqlite-vec` `vec0` virtual table 参与检索。
- 不同 `embedding_model`、`embedding_version` 或 `embedding_dim` 的向量不能混在同一次相似度检索中。
- `embedding_type` 至少区分 `face`、`scene`；v0.3 默认不启用 `appearance` 身份匹配。若后续显式开启 appearance 降级，必须使用单独 `embedding_type="appearance"`、更高阈值，并且不和 `face` embedding 混查。
- 第一版使用 `sqlite-vec` 做向量 KNN / distance 查询，不在业务代码里手写 embedding search。
- `store.py` / `retriever.py` 暴露小接口，避免业务层依赖 `sqlite-vec` SQL 细节。
- DB、模型、cache、runtime artifacts 不进入 Git。

## 7. Embedding Backend

接口只表达产品需要：

```text
embed_person(image_crop) -> EmbeddingResult
embed_scene(image_or_crop) -> EmbeddingResult
```

backend 模式：

| 模式 | 用途 |
| --- | --- |
| `disabled` | 默认安全关闭 memory 识别，只允许服务启动 |
| `fake` | 单元测试和 PC E2E，输出 deterministic embedding |
| `local` | 显式配置本地 face / scene embedding 模型后启用 |

原则：

- 不训练模型。
- 不隐式联网下载。
- 不把模型 cache 写入系统或用户目录。
- 人脸身份和场景语义是两类不同问题，可以在同一个 `MemoryEmbeddingBackend` 接口下使用两个本地模型；不要强行用一个模型解决所有问题。

### 7.1 技术选型结论

小规模商店门口应用优先使用内嵌 SQLite + `sqlite-vec` 完成 metadata、向量存储和检索：

- v0.3 默认引入 `sqlite-vec` 作为唯一向量检索依赖；不引入 FAISS、Qdrant、Postgres 或独立向量服务。
- 选择 `sqlite-vec` 的原因是数据量会较快增长，且不希望在项目业务代码中维护 embedding search、distance 计算和后续性能优化。
- `sqlite-vec` 与 SQLite 同进程运行，仍然保持单文件 DB、简单部署和本地化数据边界。
- Python 侧通过 `sqlite_vec.load()` 加载扩展；release runtime 必须显式包含 `sqlite-vec` wheel 或预编译 extension，不允许运行时联网下载。
- `retriever.py` 只封装一种检索实现：SQLite metadata filter + `sqlite-vec` vector query。不要同时保留 Python 手写 cosine search 作为第二套主路径。
- 检索时按 `embedding_type`、`embedding_model`、`embedding_version`、`embedding_dim` 过滤后，再执行向量查询，避免不同模型向量混搜。
- 预期数据量是单店从几百到数万条 embedding。该规模下 `sqlite-vec` 比手写线性检索更适合作为默认实现，同时仍比独立向量数据库简单。
- SQLite DB 是 runtime data，放在 `runtime/memory/`，不进入 Git。备份和清理用普通文件级操作，不建设数据库服务。
- 如果 `sqlite-vec` 在某个目标平台不可用，服务端必须 fail fast，并明确提示缺少 `sqlite-vec` runtime；不要自动降级到另一套手写搜索逻辑。

模型选择：

| 用途 | v0.3 选择 | 理由 |
| --- | --- | --- |
| 已知人物身份 | face detector + face embedding local model | 人物身份需要脸部特征；YOLO person/pose 特征不能可靠区分身份 |
| 场景记忆 | scene visual embedding local model | v0.3 只对整图场景做图像相似度；bbox/point/region 场景目标留到查询侧支持 region query 后开放 |
| 当前实时检测/追踪 | 继续使用 YOLOv8n-pose | 只负责 person bbox、pose、tracking、event、gaze，不承担长期身份 |

YOLO latent / embedding 的边界：

- Ultralytics 支持从指定层导出 embedding，可用于聚类或相似度探索。
- 但 v0.3 不把 YOLO pose/detection latent 作为正式人物身份 embedding。YOLO 当前主模型是为 detection/pose 训练，特征不保证跨衣着、角度、遮挡和模型版本稳定，也不是 face identity embedding。

本地模型 bundle 合同：

- face 主线要求 SCRFD detector 9-output/kps 合同；detector 输入使用等比例 resize + pad/letterbox，bbox/kps 用同一组 scale/pad 反解回原图。
- scene 主线要求 `preprocess.resize_mode = "resize_shorter_center_crop"`，`input_size` 必须是 `[width, height]` 两个正整数；不接受 1D 或 NCHW 形状猜测。
- YOLO latent 可以作为后续研发 spike，用于粗粒度区域相似或调试，不进入 v0.3 交付验收。
- confirmed `known_person_present` 只基于 face embedding；没有可用人脸时不输出已知人物事件。

参考资料：

- Ultralytics 支持 embedding / feature extraction：<https://docs.ultralytics.com/modes/predict/>
- InsightFace / ArcFace face recognition 体系：<https://github.com/deepinsight/insightface>
- OpenCLIP image/text embedding 体系：<https://github.com/mlfoundations/open_clip>
- SQLite BLOB / embedded database：<https://www.sqlite.org/datatype3.html>
- sqlite-vec SQLite vector extension：<https://github.com/asg017/sqlite-vec>
- sqlite-vec Python 使用方式：<https://alexgarcia.xyz/sqlite-vec/python.html>

## 8. 低频事件与上下文

新增 server semantic events：

| 事件 | 触发 |
| --- | --- |
| `known_person_present` | 当前稳定人物匹配到已教学 `person_profile`，且达到 threshold / margin / cooldown |
| `scene_activated` | 当前整图匹配到已教学 scene memory，且达到 threshold / cooldown |

事件示例：

```json
{
  "type": "semantic_event",
  "event_id": "front:mem_evt_000001",
  "event": "known_person_present",
  "camera": "front",
  "track_id": 7,
  "confidence": 0.86,
  "duration_ms": 0,
  "lifecycle_state": "confirmed",
  "evidence": {
    "memory_match_id": "match_000001",
    "matched_type": "person",
    "matched_id": "person_000001",
    "embedding_id": "emb_face_000001",
    "match_type": "face",
    "match_score": 0.86,
    "top2_margin": 0.09,
    "source_target_mode": "track_id",
    "source_frame_id": 1234,
    "source_frame_timestamp_ms": 1780000000000,
    "embedding_model": "local-face-v1"
  },
  "memory_context": {
    "person": {
      "person_id": "person_000001",
      "display_name": "张三",
      "description": "店长，熟悉新品陈列和现场活动",
      "tags": ["staff", "manager"],
      "match_confidence": 0.86
    },
    "conversation_summaries": [
      "上次问过新品尺码，偏好浅色外套。"
    ]
  },
  "text": "看到已知人物：张三"
}
```

`scene_activated` 使用同一个 `semantic_event` 合同，最小 evidence：

```json
{
  "memory_match_id": "match_000002",
  "matched_type": "scene",
  "matched_id": "scene_000001",
  "embedding_id": "emb_scene_000001",
  "match_type": "scene",
  "match_score": 0.82,
  "top2_margin": 0.07,
  "source_target_mode": "scene",
  "source_frame_id": 1238,
  "source_frame_timestamp_ms": 1780000000400,
  "embedding_model": "local-scene-v1"
}
```

CLI 行为：

- `BOTIFIED_ALLOWED_EVENTS` 增加 `known_person_present`、`scene_activated`。
- CLI 不复算 memory 规则，只按 event、event_id、track_id、`memory_context` 做 notification gate。
- Botified 顶层字段不变，仍把 compact JSON 放在 `request` 中的 `visual_context=` 后面。
- `visual_context.memory_context` 只保留 agent 需要的短背景，不包含图片、crop、embedding 或完整 track 列表。
- memory events 仍遵守现有 `semantic_event` 合同，必须带 `lifecycle_state="confirmed"` 和最小 `evidence`。`memory_match_id` 是后续纠错使用的稳定引用，不代表用户身份。

## 9. 配置

只增加必要配置：

```toml
[memory]
enabled = false
db_path = "runtime/memory/visual_memory.sqlite3"
frame_cache_seconds = 5
query_interval_ms = 1000
queue_size = 2

[memory.embedding]
backend = "disabled" # disabled | fake | local
person_model_path = ""
scene_model_path = ""

[memory.matching]
known_person_threshold = 0.82
known_person_margin = 0.06
scene_threshold = 0.78
event_cooldown_ms = 60000
```

不要为每个事件、每个字段增加大量配置项。真实阈值先通过 `val-data` 和手工 replay 调整。

## 10. 开发步骤

1. 定义 memory API request / response dataclass 和 schema contract。
2. 在 `app.state.memory_service` 接入 app-level recent frame cache，保存每个 camera 的最近帧、connection id、frame id、timestamp、JPEG bytes 和 compact visual_state 摘要。
3. 实现 target resolver，覆盖 `scene`、`attention_target`、`track_id`、`bbox`、`point_uv`。
4. 实现 SQLite store、`sqlite-vec` vector table 和向量查询。
5. 实现 `MemoryEmbeddingBackend` 的 `disabled` 和 `fake` 模式，先把测试闭环跑通。
6. 实现 `teach_person`、`teach_scene`、conversation summary 和 external user link API。
7. 实现低频 retriever 和 memory event generator。
8. 在 `visual_state.semantic_events` 中串接 `known_person_present`、`scene_activated`。
9. 更新 CLI Botified mapper allowlist 和 `visual_context.memory_context` 投影。
10. 接入真实 local embedding backend；模型路径显式配置，模型文件不进 Git。
11. 编写一个开发测试工具 `tools/run_memory_e2e.py`，先用 fake embedding 和 `val-data` 做 teach + replay 侧链验证；local backend 完成后再作为真实模型 smoke 的入口。
12. 跑现有 PC local E2E，确认 v0.2 主链路不退化。

## 11. 测试计划

测试只覆盖核心功能和高风险集成，不测试测试工具本身。

### 11.1 单元测试

- target resolver：`attention_target`、`track_id`、`bbox`、`point_uv`、过期帧和无目标错误。
- store：profile、scene、summary、external link 的写入和查询。
- vector search：`sqlite-vec` 查询、threshold、top1/top2 margin、低相似度不返回。
- runtime dependency：`sqlite-vec` 可加载；缺失时 fail fast 并给出明确错误。
- memory events：cooldown、同一人同一事件不刷屏。
- memory event delivery：worker 完成结果只追加到后续同 camera 新鲜 `visual_state`，过期结果不补发。
- CLI formatter：`memory_context` 被投影进 `visual_context`，不改变 Botified 顶层字段。

### 11.2 集成测试

- fake embedding + tmp SQLite：teach person 后 query 能返回同一 `person_id`。
- fake embedding + tmp SQLite：teach scene 后相似场景能返回 `scene_id`。
- app-level cache：无活跃 stream、帧过期、旧 `track_id` 或 camera 不匹配时，teach API 返回错误且不写入。
- 低相似度人物/场景不触发 confirmed event。
- conversation summary 绑定到 `person_id` 后，`known_person_present` 能返回短摘要。
- `external_user_ref` 绑定后，agent 查询外部用户能拿到同一 person profile。
- embedding backend 慢或失败时，不阻塞主 `process_frame()` 返回。

### 11.3 PC 本地 E2E

必须使用 `val-data`：

- 从 `val-data` 某个场景的第 N 帧 teach person，再 replay 同场景后续帧，验证 `known_person_present`。
- 从 `val-data` 某个场景 teach scene，再 replay 同场景，验证 `scene_activated`。
- 用不同 scene 作为负例，验证不会高置信误触发。
- 开启 memory 后继续跑现有 `tools/run_cli_local_e2e.py --full-scene --all-scenes --head-state stationary`，验证 DDS/CLI/server/Botified/gaze 主链路不退化。
- fake embedding 和 `tools/run_memory_e2e.py` 默认命令用于手工确定性检查，不是默认发布 gate：

```bash
uv run python tools/run_memory_e2e.py --data-dir val-data --out artifacts/memory-e2e --scene pic_hello
```

- 真实交付还必须在 `local` backend + 显式模型路径下手工跑一次 `val-data` teach / replay smoke；该 local smoke 不加入默认发布 gate。只有报告满足 `ok=true`、`uses_real_model_backend=true`、v0.3 `known_person_present` + `scene_activated` assertions 通过时，才能把 local backend 视为 PC 本地可用：

```bash
uv run python tools/run_memory_e2e.py \
  --data-dir val-data \
  --out artifacts/memory-e2e-local \
  --scene pic_hello \
  --embedding-backend local \
  --person-model-path runtime/models/<face-model-bundle> \
  --scene-model-path runtime/models/<scene-model-bundle>
```

`val-data/`、runtime DB、模型、cache 和 artifacts 仍然不进 Git。

## 12. 验收标准

v0.3 交付完成必须满足：

- `memory.enabled=false` 时，现有 v0.2 PC local E2E 行为不变。
- `memory.enabled=true` 且 fake embedding 下，teach person / replay / Botified `known_person_present` 完整跑通。
- `memory.enabled=true` 且 fake embedding 下，teach scene / replay / Botified `scene_activated` 完整跑通。
- `memory.enabled=true` 且 local embedding backend 配置了显式模型路径时，能完成一次 `val-data` teach / replay smoke，且报告满足 `ok=true`、`uses_real_model_backend=true`、v0.3 `known_person_present` + `scene_activated` assertions 通过；server 不隐式下载模型。
- 低相似度人物和场景不会输出 confirmed memory event。
- `known_person_present` 和 `scene_activated` 受 cooldown 和 CLI notification gate 控制，不每帧唤醒 agent。
- Botified frame 顶层字段不变，`visual_context.memory_context` 可解析且足够短。
- external user link 查询能从 `external_user_ref` 找回 `person_id` 和元信息。
- conversation summary 能随人物识别返回给 agent。
- memory worker backlog 有上限；embedding 慢时只丢弃或延迟 memory 事件，不拖慢 10Hz gaze。
- PC 本地 current gate 仍能报告 PC-simulated pass；不能声称真机、RK3588 或现场已验证。

## 13. 风险与控制

| 风险 | 控制 |
| --- | --- |
| 认错人导致 agent 说错话 | threshold、top1/top2 margin、unknown fallback、低频 cooldown |
| 教学目标选错 | target resolver 返回质量和错误原因；不确定时不写入 |
| memory 拖慢实时链路 | bounded queue、低频 query、异步 worker、主链路不等待 |
| payload 变大影响 agent | `memory_context` 只投影短 profile 和短摘要 |
| 模型和 cache 污染环境 | uv 管理；显式 runtime/model path；不写系统或用户目录 |
| 功能膨胀 | v0.3 不做匿名熟人、复杂手势、纠错系统、后台和云同步 |

## 14. Review 结论

产品和研发 review 后的结论：

- v0.3 的最小可交付价值是“显式教学后可识别”，不是“自动理解所有人和所有场景”。
- 该版本必须把 memory 做成 server 内低频侧链，不能影响 10Hz detection / tracking / gaze。
- SQLite + `sqlite-vec` 足够支撑第一版，不应把独立向量数据库和治理功能提前变成主线。
- 计划已经覆盖用户提出的核心闭环：教人、教场景、看到认识的人通知 agent、返回短背景、外部消息用户显式绑定。
- 自动熟人、泛化自然指向和复杂纠错进入 v0.4；受限的 person-to-person introduction 已由 stable memory/teaching GA 计划覆盖。
