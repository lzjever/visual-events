# Visual Events v0.2 商店门口事件质量开发计划

日期：2026-06-28

## 1. 版本定位

v0.2 的目标是把 v0.1.0 已经跑通的“看到人、追踪人、输出事件和 gaze”的能力做稳，服务商店门口揽客场景。

本版本不追求新增一大批事件类型，也不把 Visual Memory / RAG 做进主线。下一版最有价值的事情是：

- 减少重复通知和事件刷屏。
- 让 passing by、approaching、stopped、waving 的触发更稳。
- 让 agent 收到事件时知道“为什么触发”，而不是只有一个事件名。
- 在多人、短遮挡、track id 切换时更稳定地选择和保持交互目标。
- 保持 PC 本地 10Hz 端到端验证通过，并不恶化未来 RK3588 迁移路径。

一句话：v0.2 是 `Storefront Engagement Stability`，不是记忆系统、身份系统或视觉平台扩张。

## 2. 产品结论

推荐主线功能：

1. 事件生命周期：对外给 confirmed 事件补 `lifecycle_state`，内部只在能简化现有 active/cooldown 逻辑时收拢状态，不做通用生命周期框架。
2. 事件 evidence：每个语义事件都带可机读证据，方便 agent 判断、回放验证和现场调参。
3. 最小 scene context：在 `visual_state` 中告诉 agent/调试工具此刻是否有可交互目标，以及为什么暂时不适合交互。
4. 短遮挡与 track 切换去重：短时间 lost / 换 id 时不要把同一个人当新人重复输出同类视觉事实。
5. 注视目标稳定性诊断：保持高频 gaze DDS 设计不变，最大稳定人物只是默认 baseline；事件目标与 attention target 是否一致必须在 `visual_context.event_target.matches_attention_target` 中暴露。
6. Agent 视觉快照：CLI 在 Botified 通知中附带一份小的当前画面摘要，让 agent 知道事件发生时的环境构成。
7. 可视化调试增强：扩展当前 replay HTML，能肉眼看到事件证据、scene context 和 reacquire 信息。

### 2.1 产品行为矩阵

Visual Events server 只输出视觉事实和 evidence，不替 agent 决定机器人是否开口、说什么、说几次。但 agent 每轮成本和延迟都高，所以 agent 端 CLI 在输出 Botified 前必须有一个很小的 notification gate：它只决定“是否值得唤醒 agent”，不决定“agent 应该怎么做”。

| 场景 | server semantic event | CLI notification gate 默认处理 | scene context |
| --- | --- | --- | --- |
| 无人 | 无 | 不通知 | `no_target` |
| 刚稳定进入画面 | `person_appeared` | 默认不通知，除非后续产品明确要唤醒 | `available` 或 `no_engage_target` |
| 快速横穿 | `person_passing_by`，evidence 标明速度快 | 不通知 | `no_engage_target`，reason=`passing_fast` |
| 慢速路过且近区稳定 | `person_passing_by` | 可通知，但允许短延迟合并 | `available` |
| 朝机器人靠近 | `person_approaching_robot` | 可通知，但允许短延迟合并 | `available` |
| 靠近后停下 | `person_stopped_near_robot` | 通知，优先于 pending approaching/passing | `available` |
| 挥手 | `person_waving` | 高优先级，尽量立即通知 | `available` |
| 离开 | `person_left` | 默认不通知；如果离开的是最近通知过或 in-flight 的目标，则发送一次结束事实 | `no_target` 或 `no_engage_target` |

CLI notification gate 的规则：

- server 仍持续处理 10Hz 视觉流，`visual_state.semantic_events` 只表达 confirmed 视觉事实。
- CLI 只把经过 notification gate 的 allowlist 事件输出成 Botified frame。
- CLI 不复算 passing/approaching/stopped/waving 规则；它只使用 server 给出的 event type、evidence、scene_context、track_id 和 event_id 做通知节流/合并。
- CLI pending queue 的 key 优先使用 server evidence 中的 `runtime_person_slot`；没有该字段时使用 `track_id`；短遮挡 reacquire 后如果 event evidence 或 `scene_context.target_reacquired` 提供 `reacquired_from_track_id`，CLI 必须把新旧 track 映射到同一个 pending key。
- 对低优先级但可能很快升级的事件做短 coalescing：`passing_by` / `approaching` 可等待 500-1000ms；如果期间收到 `stopped` 或 `waving`，只通知更高价值的新事实。
- `person_waving` 不等待低优先级窗口，除非全局 rate limit 正在保护系统。
- CLI 维护本地 in-flight key：Botified 输出后，在 `timeout_secs` 内视为该 key 正在被 agent 处理；没有 agent ack 输入时用 TTL 作为保守 backpressure，有 ack/status 通道后再消费真实状态。
- `person_left` 对普通路人默认静默；对 in-flight key 或最近通知过的 key 允许通知一次离开事实，让 agent 停止继续面向已离开的对象。
- server 内部可以使用短期 runtime `person_slot` 来做视觉事实去重；它只存在于当前进程内，不写盘，不跨会话，不代表身份。
- server 负责同一 track/person slot 的同一 event type rising edge 和 cooldown。
- 短遮挡或换 id 被 reacquire 为同一 runtime person slot 时，server 不重复输出 `person_appeared` 这类重复事实。
- CLI 不因为同一个人已经通知过 `approaching` 就永久禁止后续 `stopped` 或 `waving`；但通知之间应受最小间隔和 rate limit 保护，避免连续昂贵 agent turn。
- CLI 的全局 burst/rate limit 只保护 agent 成本和系统稳定，不表达具体揽客动作策略。

明确不做：

- 不训练新模型。
- 不引入 ReID、人脸识别、长期人员身份或数据库。
- 不把 CLI 变成事件判断方。
- 不让 CLI 复算视觉事件；CLI 只做 agent 通知成本控制。
- 不把 10Hz 高频 `visual_state` 发给 Botified。
- 不让服务端或 CLI 直接操纵运控。
- 不建设后台、看板、交付审计系统或复杂报告。
- 不把真机、RK3588 或现场测试作为 v0.2 通过条件；v0.2 gate 仍是 PC 本地模拟跑通。

隐私/合规不是当前私有场景的产品阻塞点，但这不是 v0.2 延后人脸、身份和长期记忆的原因。真正原因是这些能力会新增模型、存储、错误恢复和实时链路风险，不符合本版本 KISS/YAGNI。

## 3. 当前技术判断

当前主链路是：

```text
JPEG frame
  -> server inference
  -> tracker
  -> attention selector
  -> event engine
  -> visual_state @10Hz
  -> CLI gaze DDS + low-frequency Botified event
```

现有 `YOLOv8n-pose + tracker + rules` 足够支撑 v0.2 主线：

- pose 模型输出 person bbox 和人体关键点，可支撑 person detection、最大人物、头部注视点和 waving 规则。
- tracker 已输出 bbox、area、center、head、velocity、age、lost、confidence 等状态，足够做 passing by、approaching、stopped 和短遮挡去重。
- RK3588 未来兼容要求反而支持“少模型、少依赖”的方向。v0.2 不新增实时模型，避免把未来 RKNN 迁移变复杂。
- `evidence` 和 `scene_context` 只能使用现有 `InferBackend` 输出、tracker 输出和 Python 几何/时间窗口计算；不能引入新的实时模型、GPU-only 依赖，也不能把 RKNN 适配耦合进 CLI。

Visual Memory / Scene-Grounded RAG 技术上可行，但不是同一层能力：

- 它不应该进入 `/v1/stream` 每帧实时处理路径。
- 它需要另开计划和资源，不影响 v0.2 pass/fail。

参考事实：

- Ultralytics pose 模型输出 bbox/keypoints，适合当前人体关键点规则链路：<https://docs.ultralytics.com/tasks/pose/>
- Ultralytics 已提供 Rockchip RKNN 导出路径，未来 RK3588 迁移可以沿模型 backend 替换推进：<https://docs.ultralytics.com/integrations/rockchip-rknn/>
- 视觉记忆候选细节保留在 `docs/product-candidate-visual-memory-rag.md`，不进入本计划主线。

## 4. v0.2 功能范围

### 4.1 事件生命周期

对外协议只承诺 confirmed 事件。内部生命周期是实现手段，不是新框架：

```text
candidate -> confirmed -> active -> cooldown -> ended
```

对外行为保持简单：

- `candidate`、`active`、`cooldown`、`ended` 不输出给 Botified。
- `semantic_events[]` 只输出 confirmed 事件。
- 输出事件中增加 `lifecycle_state: "confirmed"`，用于协议自描述和测试。

首批纳入生命周期管理的事件：

- `person_appeared`
- `person_left`
- `person_passing_by`
- `person_approaching_robot`
- `person_stopped_near_robot`
- `person_waving`
- `attention_target_changed`

实现要求：

- 所有事件仍在 server 的 `EventEngine` 中判断。
- 每个事件只保留一套规则，禁止 CLI、processor 或测试工具重复实现业务判断。
- 不做通用规则插件框架。
- 只有当小的 condition state 表能替代并简化现有 `_active_conditions` 时才引入；否则先沿用现有 proposal/active/cooldown 流程。

### 4.2 事件 evidence

每个 semantic event 增加 `evidence`。字段必须来自当前 track/history/attention/机器人相机运动状态，不能来自推测身份或长期记忆。

文档中的 `camera_motion_state` 指机器人头部或相机是否正在运动，来源是当前协议里的 `FrameMessage.head_motion_state`。它不是人的头部朝向，不做人脸朝向、人眼 gaze 或“人是否看向机器人”的判断。

示例：

```json
{
  "type": "semantic_event",
  "event_id": "front:evt_000456",
  "event": "person_approaching_robot",
  "camera": "front",
  "track_id": 7,
  "confidence": 0.82,
  "duration_ms": 1200,
  "lifecycle_state": "confirmed",
  "evidence": {
    "bbox_area_ratio_start": 0.08,
    "bbox_area_ratio_end": 0.15,
    "area_delta": 0.07,
    "area_growth_ratio": 1.88,
    "visible_duration_ms": 1800,
    "camera_motion_state": "stationary"
  },
  "text": "有人正在靠近机器人"
}
```

字段原则：

- 数值必须是有限值，不输出 NaN/Inf。
- 每类事件只放最关键的 3-8 个字段。
- evidence 不用于生成长篇解释，不做审计报告；它是调试和 agent 决策的简短事实。

建议 event-specific evidence：

| 事件 | 最小 evidence |
| --- | --- |
| `person_appeared` | `runtime_person_slot`、`visible_duration_ms`、`bbox_area_ratio`、`salient_reason` |
| `person_left` | `runtime_person_slot`、`lost_duration_ms`、`last_bbox_area_ratio` |
| `person_passing_by` | `runtime_person_slot`、`dx_ratio`、`avg_vx_px_s`、`crossed_side_bands`、`camera_motion_state`、`passing_speed_class` |
| `person_approaching_robot` | `runtime_person_slot`、`bbox_area_ratio_start`、`bbox_area_ratio_end`、`area_growth_ratio`、`area_delta`、`camera_motion_state` |
| `person_stopped_near_robot` | `runtime_person_slot`、`bbox_area_ratio`、`speed_px_s_p95`、`stationary_duration_ms`、`camera_motion_state` |
| `person_waving` | `runtime_person_slot`、`wrist_x_span_px`、`wrist_x_span_bbox_ratio`、`wrist_y_relative_to_shoulder_px`、`wave_duration_ms`、`keypoint_min_confidence` |
| `attention_target_changed` | `previous_track_id`、`target_track_id`、`switch_reason` |

这些 key 是 v0.2 的 event-specific required evidence keys，EventEngine unit tests 和 replay accumulator 必须验证字段存在、类型正确、数值有限。`runtime_person_slot` 只在当前 server runtime 内有效，用于 CLI notification gate 的短期合并；不写盘，不跨连接，不代表身份。

### 4.3 最小 scene context

在 `visual_state` 中增加 `scene_context`，用于每帧高频状态和调试。默认不进入 Botified。

建议 schema：

```json
{
  "scene_context": {
    "engagement_state": "available",
    "attention_available": true,
    "target_track_id": 7,
    "no_engage_reasons": [],
    "target_reacquired": null
  }
}
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `engagement_state` | `available`、`no_target`、`no_engage_target` 三选一 |
| `attention_available` | 是否有稳定且可用于 gaze 的目标 |
| `target_track_id` | 当前可交互目标；无目标时为 `null` |
| `no_engage_reasons` | 有人但视觉上不适合互动的原因，例如 `too_far`、`passing_fast`、`unstable`、`outside_engagement_roi`、`ignored_zone`、`camera_motion_not_stationary` |
| `target_reacquired` | 短遮挡恢复信息；无恢复时为 `null` |

控制原则：

- scene context 是状态，不是事件刷屏入口。
- v0.2 不新增 `scene_attention_available` 这类 Botified 事件；先留在 `visual_state` 和可视化工具里。
- 如果以后要把 scene transition 发给 Botified，必须单独证明不会刷屏。

### 4.3.1 Agent 视觉快照

server 已经通过 10Hz `visual_state` 提供当前画面的全局信息，包括 tracks、person count、attention target、scene flags 和 v0.2 新增的 scene context。agent 不应该接收 10Hz 视觉流，但在 CLI 决定输出 Botified frame 时，应该附带一份小的 `current_scene` 快照。

目的：

- 让 agent 理解事件发生时的环境，而不是只看到一个事件名。
- 降低 agent 二次询问或误判成本。
- 保持高频视觉状态和低频 agent turn 分离。

Botified 当前只允许 `id`、`urgency`、`timeout_secs`、`request`、`expect` 这几个字段。为避免同时修改 Botified core，v0.2 不新增 Botified 顶层字段；CLI 把小型 `visual_context` 作为紧凑 JSON 片段放进 `request` 文本。未来如果需要真正的顶层结构化字段，必须另行修改 Botified core/parser/docs/tests。

```json
{
  "id": "visual:front:evt_000456",
  "urgency": "normal",
  "timeout_secs": 8,
  "request": "视觉事件：有人正在靠近机器人。event=person_approaching_robot track_id=7。visual_context={...紧凑JSON...}。请根据当前上下文决定是否回应；处理完成后回复 ack。",
  "expect": "ack"
}
```

`request` 中的 `visual_context` JSON 内容：

```json
{
  "visual_context": {
    "event_target": {
      "track_id": 7,
      "runtime_person_slot": 3,
      "visible_now": true,
      "matches_attention_target": true,
      "event_age_ms": 120,
      "position": "center",
      "size": "near",
      "bbox_area_ratio": 0.15
    },
    "trigger_evidence": {
      "duration_ms": 1200,
      "bbox_area_ratio_start": 0.08,
      "bbox_area_ratio_end": 0.15,
      "area_growth_ratio": 1.88
    },
    "current_scene": {
      "camera": "front",
      "frame_age_ms": 120,
      "person_count": 2,
      "attention_target": {
        "track_id": 7,
        "position": "center",
        "size": "near",
        "bbox_area_ratio": 0.15,
        "center_uv": [420.0, 360.0]
      },
      "other_people_count": 1,
      "engagement_state": "available",
      "no_engage_reasons": []
    }
  }
}
```

规则：

- CLI 使用最近一帧 fresh `visual_state` 生成 `visual_context.current_scene`，超过 stale 阈值则省略或标记 `stale=true`。
- `request` 中固定使用 `visual_context=<minified-json>` 格式；formatter 负责输出紧凑 JSON，stdout parser / PC oracle 用 marker 后的 brace-balanced JSON 提取，不靠自然语言分词猜测。
- `visual_context.event_target` 必须描述触发事件的人，而不只是 attention target；多人场景下必须包含 `matches_attention_target`。
- `visual_context.trigger_evidence` 是 event evidence 的小型白名单投影，最多 3-6 个字段；不要把完整 debug evidence 倾倒给 agent。
- `person_count` 来自 server tracks / scene flags。
- `position` 是粗粒度区域，例如 `left`、`center`、`right`，不要传复杂几何解释。
- `size` 是距离近似，例如 `far`、`mid`、`near`，由 `bbox_area_ratio` 或 bbox height ratio 推导。没有标定或深度模型时，不输出真实米制距离。
- 最多摘要 attention target 和少量人数统计，不把全部 10Hz track 列表塞进 Botified。
- 不发送图片、crop、embedding、人脸身份或长期记忆。
- `common/schema/protocol.md`、Botified request samples、formatter tests、stdout parser 和 PC oracle 必须同步这个 request 内嵌 `visual_context` 合同。

这份快照对商店门口 agent 很重要：它可以判断“眼前是否多人”“目标是否靠近”“事件目标是不是当前注视目标”“是否只是快速路过”。但它仍然只是上下文，最终回应动作由 agent 自己决定。

### 4.3.2 门口区域配置

商店门口场景需要一个很小的画面区域配置，避免把背景路人、店内远处人物或橱窗外横穿误当成揽客目标。

配置保持简单：

- `engagement_roi_norm`：归一化矩形或多边形，坐标范围 `[0,1]`，默认整幅画面。
- `ignore_rois_norm`：归一化矩形列表，坐标范围 `[0,1]`，默认空。
- ROI 只影响事件/scene context 的可交互判断；tracks 仍保留在 `visual_state`，方便调试。
- ROI 外的人可进入 `scene_context.no_engage_reasons=["outside_engagement_roi"]`；ignore ROI 内的人使用 `ignored_zone`。
- 不做地图、标定、3D 区域或复杂场地建模。

### 4.4 短遮挡与重复事实抑制

目标：同一个人短暂遮挡、短暂 lost 或 tracker 换 id 后，不重复触发 `person_appeared` 或同一类重复事实事件。

保守规则：

- 默认 `reacquire_alias_window_ms` 收敛到 1200ms；允许调参范围 500-1500ms。
- bbox 中心距离不超过图像对角线的 8%。
- 新旧 bbox 面积比例在 0.5-2.0 之间。
- 若新旧速度都可用，方向夹角不能明显相反；速度缺失时不因方向项失败。
- reacquire 只用于短期去重，不生成长期身份。

输出要求：

- 可在 event evidence 或 `scene_context.target_reacquired` 中记录 `reacquired_from_track_id`。
- reacquire evidence 至少包含 `reacquire_elapsed_ms`、`reacquire_center_distance_px`、`reacquire_area_ratio`。
- 无法确定同一物理人时，不伪造身份；只使用更保守的 cooldown。

### 4.5 注视目标稳定性

高频 gaze 仍然按当前设计：

- server 输出 `attention.target_track_id` 和 `target_uv`。
- CLI 只发布 DDS gaze target，不计算头控指令，不直接操纵运控。
- Botified 不接收 10Hz 高频视觉流。
- 默认 attention 策略仍是最大且稳定的人，这是 v0.2 的 gaze baseline，不等于“agent 应该回应的人”。
- 当事件目标与 attention target 不一致时，不在 CLI 内修正 gaze；通过 `visual_context.event_target.matches_attention_target=false` 明确暴露给 agent 和调试工具。

v0.2 只增强诊断：

- 在 attention 或 scene context 中补充可选 `target_dwell_ms`。
- replay 统计 attention switch 次数、lost hold、最大人物不一致帧数。
- 可视化页面显示当前 gaze 点和目标 track。

不要把 attention selector 变成“要不要打招呼”的业务引擎。视觉可交互状态和事实判断仍在 `EventEngine` / scene context；agent 动作策略不在本 repo。

### 4.6 可视化调试增强

扩展 `tools/visualize_service_replay.py`，只作为随手调试工具，不产品化：

- 在图片上显示 track、attention target、gaze 点。
- 显示当前帧 semantic events。
- 显示 event evidence 摘要。
- 显示 scene context：`engagement_state`、`no_engage_reasons`、reacquire。
- 输出静态 HTML 到 ignored `artifacts/`。

不做：

- 不做 Web dashboard。
- 不做前端项目。
- 不做交互式标注系统。
- 不为可视化工具扩一大套测试；只保留核心 smoke。

## 5. 实现方案

### 5.1 模块边界

| 模块 | v0.2 改动 |
| --- | --- |
| `src/visual_events_server/events/engine.py` | 主改动点：生命周期、evidence、scene context、短遮挡去重 |
| `src/visual_events_server/processor.py` | 只串接 `semantic_events` 和 `scene_context` 到 `visual_state` |
| `src/visual_events_server/config.py` / config samples / config tests | 增加 `engagement_roi_norm`、`ignore_rois_norm` 解析与校验，并传入 EventEngine 或 SceneContext 配置 |
| `src/visual_events_server/attention/selector.py` | 可补轻量诊断，不做事件规则 |
| `src/visual_events_server/tracking/byte_tracker.py` | 原则上不改；保持纯 tracker |
| `src/visual_events_cli/*` | 不新增视觉识别规则；增加 Botified notification gate、coalescing、rate limit、`visual_context` 快照和 allowlist 透传 |
| `common/schema/protocol.md` / `common/schema/samples/*` / `tests/contract/*` | 同步 v0.2 `visual_state`、semantic event evidence、scene context 和 Botified request 内嵌 `visual_context` 合同 |
| `tools/visualize_service_replay.py` | 调试显示增强 |
| `tools/run_val_data_e2e.py` | 增加最少必要检查：evidence、scene context、重复事件抑制 |
| `tools/replay_val_data.py` / `tools/run_cli_local_e2e.py` | 同步 Botified oracle 到 CLI notification gate 语义，不再按旧事实事件列表要求所有 allowlist 事件都唤醒 agent |

### 5.2 EventEngine 设计

建议最小接口和内部结构：

```text
EventEngineResult
  semantic_events
  scene_context

_EventProposal
  event
  track_id
  confidence
  duration_ms
  evidence
  lifecycle_state = "confirmed"

_ConditionState
  slot_id
  event
  state
  first_seen_ms
  confirmed_ms
  last_seen_ms
```

实现原则：

- `EventEngine.update()` 从 `list[dict]` 改为返回 `EventEngineResult`；`processor` 只读取 `result.semantic_events` 和 `result.scene_context`。
- `_EventProposal` 由每个现有规则产生，携带 evidence。
- `_emit_if_allowed()` 统一写 event schema、cooldown、finite evidence 和 event id。
- `_active_conditions` 只有在能减少现有分散状态时才替换成 `_ConditionState`；不要为了生命周期图新建框架。
- runtime `person_slot` 去重放在 `EventEngine`：只用于 rising edge、cooldown、短遮挡 reacquire 和同类重复事件抑制，不限制不同事实事件的上报。
- person slot / reacquire 保持在 `EventEngine`，不下沉 tracker。
- `reacquire_alias_window_ms` 从当前偏长窗口收敛到短遮挡窗口；server cooldown 只处理同类视觉事实的 rising-edge 去重。长期 Botified 通知成本由 CLI notification gate 的 coalescing/rate limit 控制。

### 5.3 visual_state 协议

`build_visual_state()` 增加 `scene_context` 参数：

```text
build_visual_state(..., semantic_events=events, scene_context=scene_context)
```

兼容要求：

- 旧字段不删除。
- `semantic_events` 仍为 list，空时输出 `[]`。
- `scene_context` 每帧存在，使用简单默认值：

```json
{
  "engagement_state": "no_target",
  "attention_available": false,
  "target_track_id": null,
  "no_engage_reasons": ["no_visible_person"],
  "target_reacquired": null
}
```

### 5.4 Botified / DDS

保持一个做法：

- 高频 gaze：DDS。
- 低频语义事件：Botified frame。
- CLI 只做 `event_id` 幂等和 allowlist，不做事件规则。

v0.2 不新增 Botified 事件类型，除非现有 allowlist 事件 schema 增加字段后仍兼容。

Botified 边界：

- allowlist 事件名保持当前集合：`person_appeared`、`person_left`、`person_passing_by`、`person_approaching_robot`、`person_stopped_near_robot`、`person_waving`。
- `attention_target_changed` 仍不输出到 Botified。
- CLI 不读取 evidence 做视觉规则判断；它只按 event type allowlist、event evidence、scene context、coalescing/rate limit 和 `event_id` 幂等做通知 gate 与格式化。
- 完整 `evidence` 和 `lifecycle_state` 不写入 Botified `request` 文本，避免 payload 变大；CLI 只把白名单投影写入 `visual_context.trigger_evidence`。完整字段仍保留在 `visual_state`、replay 和可视化中。

## 6. 开发步骤

### Step 1：锁住当前行为

目标：先确认现有事件、attention、协议输出没有被无意改变。

工作：

- 为 `EventEngine` 现有核心事件补少量 characterization tests。
- 为 `build_visual_state()` 补 `scene_context` 默认值协议测试。
- 明确现有 val-data 场景的 expected / forbidden 事件仍成立。
- 把产品行为矩阵写成最小测试用例：无人、快速横穿、慢速靠近、靠近后停下、挥手、多人、短遮挡。

通过条件：

- 现有单元和集成测试通过。
- CLI gaze/DDS 行为不变；Botified 行为只在 Step 6 notification gate 中改变。
- `common/schema/protocol.md`、schema samples、contract tests 和 config tests 已明确 v0.2 新增字段/配置，旧精确字段集合断言已同步更新。

### Step 2：加入 event evidence

目标：每个事件输出最小证据。

工作：

- `_EventProposal` 增加 `evidence`。
- `_emit_if_allowed()` 统一校验/清理 evidence。
- 给每类事件补最小 evidence。
- 协议测试确认每个 semantic event 都有 `lifecycle_state` 和 `evidence`。
- `tests/contract/*`、`common/schema/samples/*` 和 server fake integration tests 同步新字段集合。

通过条件：

- evidence 数值有限。
- event-specific required evidence keys 都存在，字段名与文档表一致。
- event schema 兼容现有 CLI。
- val-data replay 中所有 semantic event 都带 evidence。
- Botified request 不倾倒完整 evidence。
- `person_appeared`、`person_left`、`passing_fast` 仍可在 `semantic_events` / `scene_context` / replay 中看到事实。

### Step 3：内部生命周期状态

目标：先补 confirmed 生命周期字段和同类重复事件抑制；只有在能简化现有 active/cooldown/rising-edge 逻辑时再收拢内部状态。

工作：

- 必须先实现 `lifecycle_state="confirmed"`、rising-edge、cooldown 和短遮挡同类事件去重。
- 可选引入小的 condition state 表，但只在它能减少现有 `_active_conditions` 复杂度时做。
- 保持对外只输出 confirmed。
- 用测试覆盖 confirmed 只输出一次、cooldown 内不重复、短遮挡换 id 后同类事实不重复。

通过条件：

- 同一 track 同事件 cooldown 内不重复。
- 同类事件全局 cooldown 仍生效。
- 不因同一个人已经触发 `approaching` 就禁止后续真实发生的 `stopped` 或 `waving`。
- 机器人/相机运动为 moving/unknown 时，运动敏感事件仍被抑制。

### Step 4：短遮挡 reacquire 和重复事实抑制

目标：短遮挡或 track id 切换后不重复输出同类视觉事实，并给 CLI notification gate 足够信息做 pending 合并。

工作：

- 默认 `reacquire_alias_window_ms=1200`，允许配置 500-1500ms。
- 基于 bbox 中心、面积、lost 时间、方向做保守匹配；先用中心+面积+时间，方向只在稳定可用时参与。
- 输出 `target_reacquired`、`runtime_person_slot` 或 event evidence。
- 加 val-data 或 synthetic scene 覆盖短遮挡换 id。

通过条件：

- 短遮挡换 id 后不重复 `person_appeared` 和同类重复事实事件。
- 两个明显不同的人不会被长期合并。
- evidence 能说明发生了 reacquire 或为什么没有 reacquire。

### Step 5：scene context

目标：每帧给出简单、可解释的互动状态。

工作：

- 在 `EventEngine` 计算 `scene_context`。
- `processor` 将其写入 `visual_state`。
- 回放统计 `available/no_target/no_engage_target` 的基本分布。

通过条件：

- 有稳定近距离目标时 `engagement_state=available`。
- 有人但快速路过、太远、不稳定、在忽略区域/可揽客区域外，或机器人/相机运动非 stationary 时为 `no_engage_target`。
- 无人时为 `no_target`。
- 不产生新的 Botified 刷屏。

### Step 6：CLI notification gate

目标：减少昂贵 agent turn，不改变 server 视觉事实识别。

工作：

- 在 CLI 的 Botified mapper 中增加 pending queue。
- `person_appeared`、`person_left` 默认不通知 agent，但保留在 `visual_state`。
- `person_passing_by` 如果 evidence 显示 `passing_speed_class=fast` 或 scene context reason 含 `passing_fast`，默认不通知 agent。
- `person_passing_by` / `person_approaching_robot` 进入 500-1000ms coalescing window；窗口内收到 `person_stopped_near_robot` 或 `person_waving` 时，只通知更高价值事件。
- `person_waving` 尽量立即通知，除非全局 burst/rate limit 正在保护系统。
- 增加小的 `BotifiedNotificationConfig`，默认值先固定在 CLI 代码中：`coalesce_window_ms=800`、`same_key_min_gap_ms=8000`、`global_60s_limit=12`、`burst_1s_limit=3`。不要为每个事件开放一堆 CLI 参数。
- pending event 在任意新 `visual_state` 到达时检查 flush；coalescing 超时且 event 仍未 stale 时输出，stale 时丢弃并计数。CLI 停止前尽量 flush 非 stale pending。
- `BotifiedEventMapper` 需要可注入 `clock_ms`，用于单元测试 pending/coalescing/rate limit。
- Botified frame 的 `request` 文本内嵌小的 `visual_context` JSON：event target、trigger evidence、person count、attention target、粗粒度位置、相对大小、scene context。
- 更新 `src/visual_events_cli/botified_output.py`、Botified formatter 单测、PC E2E stdout parser：保持 Botified 顶层字段集合不变，从 `request` 中解析紧凑 `visual_context` JSON，oracle 不再只从自然语言片段解析 event/track。
- 同步更新 `tools/replay_val_data.py` / `tools/run_cli_local_e2e.py` 的 Botified oracle：低价值/可合并事件不作为 required Botified event；`person_appeared`、`person_left`、`passing_fast` 默认不要求 Botified；`approaching -> stopped` 短窗口验证最多一条高价值通知。

通过条件：

- CLI 不复算视觉事件规则，只消费 server event/evidence/scene_context。
- `person_appeared`、`person_left` 和 `passing_fast` 默认不唤醒 agent。
- `approaching -> stopped` 短窗口内最多产生一个 Botified frame，优先通知 `stopped`。
- `waving` 能升级 pending 低价值事件。
- `visual_context.current_scene` 存在且使用 fresh visual_state；stale 时明确标记或省略。
- `visual_context.event_target` 指向触发事件的人，并标明是否匹配 attention target。
- `visual_context.trigger_evidence` 包含小型触发原因，字段来自 event evidence 白名单。
- `visual_context` 不包含图片、crop、embedding、身份信息或全量 10Hz track 列表。
- PC local E2E 的 Botified oracle 与 notification gate 语义一致，不再用旧事实事件列表要求所有 allowlist 事件都唤醒 agent。
- Botified stdout 仍无 debug/status 污染。

### Step 7：可视化调试增强

目标：用户可以肉眼看懂一次 replay 为什么触发事件。

工作：

- 在 HTML 帧列表中显示 event evidence 摘要。
- 显示 scene context 和 gaze 点。
- 保持静态 HTML 输出到 `artifacts/visual-debug`。

通过条件：

- 可以用 `val-data` 中至少一个挥手/停留/经过场景生成可视化 HTML。
- 不要求产品级 UI。

### Step 8：PC 端端到端回归

目标：确认 v0.2 没破坏 v0.1.0 的可运行 baseline。

工作：

- 跑 server replay gate。
- 跑 PC local CLI E2E，使用本地 DDS 模拟输入和 gaze subscriber。
- 跑 full pytest。

权威 PC gate 命令：

```bash
uv run --group dev python tools/run_cli_local_e2e.py \
  --full-scene \
  --all-scenes \
  --head-state stationary \
  --server-config configs/pc-ga-server.toml
```

新增 `evidence`、`scene_context` 和 reacquire 检查落在现有 replay/E2E summary accumulator 中，不新建报告层、manifest 流程或审计 gate。

通过条件：

- `val-data` required scenes 全部通过。
- forbidden events 为 0。
- expected event first trigger 误差 <= 3 frames 或 <= 300ms。
- Botified 输出无重复事件风暴。
- gaze DDS 保持 10Hz 语义：tracking/lost/stale/disabled 正确。

## 7. 测试计划

TDD 只覆盖核心功能和高风险边界。`tools/run_val_data_e2e.py` 可以加最小 smoke/contract 覆盖新增 accumulator 字段，但不要扩展成测试工具测试矩阵。

### 7.1 单元测试

`EventEngine`：

- confirmed 窗口达标才输出。
- cooldown 内不重复输出。
- 同一短遮挡目标换 id 后不重复同类视觉事实。
- 机器人/相机运动为 moving/unknown 时抑制 `passing_by`、`approaching`、`stopped`。
- 同一短遮挡/reacquire 目标不重复输出同类事实事件。
- 短遮挡换 id 会保留/复用 `runtime_person_slot`；两个不同人即使中心接近，但面积比例或时间/方向不合理时不能误合并。
- 同一人后续真实发生的新事实仍允许由 server 输出，例如先 `approaching` 后 `stopped`，或停留后 `waving`。
- 每个事件 evidence 字段存在、类型正确、数值有限。
- `scene_context` 的 `available/no_target/no_engage_target` 判定。
- `engagement_roi_norm` / `ignore_rois_norm` 能把背景路人标成 `outside_engagement_roi` 或 `ignored_zone`，但 tracks 仍保留在 `visual_state`。

`AttentionSelector`：

- 最大稳定人物选择不退化。
- challenger 必须达到 switch confirm 后才切换。
- lost hold 期间 gaze 仍保持短时间有效。
- target_uv 有限且在图像范围内。

`processor/protocol`：

- `visual_state` 旧字段保持不变。
- 新增 `scene_context` 默认值。
- `semantic_events[].evidence` 不破坏 CLI/Botified contract。
- `EventEngineResult.semantic_events` 和 `EventEngineResult.scene_context` 被 processor 原样串接。

`CLI notification gate`：

- `person_appeared`、`person_left` 默认不输出 Botified。
- 已经通知过或 in-flight 的目标触发 `person_left` 时，CLI 可发送一次离开事实，帮助 agent 收尾或停止面向已离开对象。
- `passing_fast` 默认不输出 Botified。
- `passing_by/approaching` pending notification 可被 `stopped/waving` 升级，降低 Botified turn 数。
- coalescing window 之后同一人后续真实发生的新事实仍允许通知，但受最小通知间隔和全局 rate limit 保护。
- CLI pending key 使用 `runtime_person_slot`，并能用 `reacquired_from_track_id` 把短遮挡前后的 track 合并到同一个 pending key。
- CLI 不复算视觉事件规则，只消费 server event/evidence/scene_context。
- Botified `request` 内嵌小的 `visual_context` JSON，字段有限、值有限、frame fresh；stale 时不伪造当前场景。

### 7.2 集成和回放测试

必须使用 `val-data/` 做端到端验证。`val-data/` 不提交 Git。

验证内容：

- stationary full matrix：tracking、attention、events、scene context。
- 机器人/相机 moving/unknown targeted cases：运动敏感事件 suppression。
- 简单 ROI targeted cases：ROI 外/ignore ROI 内人物不触发可交互事件，但仍可调试查看 track。
- 短遮挡/换 id：重复事实和重复通知抑制。
- 事件 evidence：所有 emitted semantic event 都有可读证据。
- 可视化：至少生成一个可肉眼检查的 HTML。

### 7.3 性能回归

v0.2 不新增实时模型，因此性能目标不应变差：

- 输入/输出仍按 10Hz 设计。
- PC 本地 E2E 保持通过。
- 服务端总延迟、error rate、RSS 不明显退化。
- 未来 RK3588 迁移路径不新增实时依赖。

## 8. 验收标准

v0.2 handoff 必须满足：

- full pytest 通过。
- `val-data` replay 通过。
- PC local CLI E2E 通过。
- 所有 semantic event 有 `lifecycle_state` 和 `evidence`。
- forbidden events 为 0。
- 同一事件 cooldown 内无重复输出。
- 短遮挡/换 id 场景不重复同类视觉事实，也不重复 Botified 通知。
- `visual_state.scene_context` 每帧存在且字段稳定。
- gaze DDS 行为不变，CLI 不直接操纵运控。
- Botified 只输出通过 CLI notification gate 的 allowlist 事件，低价值/可合并事件不直接唤醒 agent。
- Botified `request` 内嵌 `visual_context`，帮助 agent 理解事件目标、触发原因、当前人数、目标位置、相对大小和 scene context。
- 可视化 HTML 能展示至少一个真实 val-data 场景的事件和证据。

产品场景验收：

| 场景 | 期望 | 禁止 | Botified |
| --- | --- | --- | --- |
| 无人 | `scene_context.engagement_state=no_target`，attention/gaze invalid 或 stale | 任何 Botified event | 无 |
| 快速横穿 | `scene_context.engagement_state=no_engage_target`，reason=`passing_fast`，server 可输出 `person_passing_by` 事实 | `person_approaching_robot`、`person_stopped_near_robot`、`person_waving` | 无 |
| 慢速近区路过 | confirmed `person_passing_by`，evidence 标明速度/近区稳定 | 同窗口误触发 `approaching/stopped/waving` | 最多 1 条，可短延迟 |
| 朝机器人靠近 | confirmed `person_approaching_robot`，面积持续增大 evidence 完整 | `person_passing_by` 同窗口抢先通知 agent | 最多 1 条，可被 stopped/waving 升级 |
| 靠近后停下 | confirmed `person_stopped_near_robot`，近距离 + 低速 + 持续时间 evidence 完整 | 远处站立被算作 stopped，approaching/stopped 短窗口双重通知 | 1 条更高价值通知优先 |
| 停下但未挥手 | `person_stopped_near_robot` 可触发，`person_waving` 不触发 | 走路摆臂误判 waving | 最多 1 条 |
| 挥手 | `person_waving`，手腕在上半身/肩部附近、有足够横向摆动和持续时间 evidence | 低置信 keypoint 单帧触发 | 优先通知，受全局 rate limit 保护 |
| 两人同时出现 | attention 选择最大且稳定目标，scene context 指向该目标 | 目标频繁切换导致 Botified burst | 受全局 burst/rate limit 保护 |
| 短遮挡/换 id | `target_reacquired` 或 event evidence 说明 reacquire，同一物理人不重复同类事实 | 创建长期身份、跨连接记忆、重复 `person_appeared` | 无重复同类通知 |
| 已通知目标离开 | server 输出 `person_left`，CLI pending/in-flight key 命中最近通知目标 | 普通背景路人离开打扰 agent | 可发送 1 条离开事实 |

所有触发 Botified 的场景都应在 `request` 中内嵌 `visual_context`；其中 `event_target`、`trigger_evidence`、`current_scene.person_count`、`current_scene.engagement_state` 是最小必填上下文。`current_scene.attention_target.position/size` 只在存在 fresh attention target 时必填；目标离开或 stale 时允许 `attention_target=null` 或省略，并由 `event_target.visible_now=false` 和 last evidence 表达离开对象。无人或 stale 场景不得伪造 fresh target。

不要求：

- 真机运行。
- RK3588 实机运行。
- 现场门店测试。
- memory/RAG 可用。
- 人脸识别或身份记忆。

## 9. Visual Memory / RAG 后续处理

Visual Memory 是有价值的未来能力，但不进入 v0.2 主线，也不作为 v0.2 通过条件。

任何 memory/RAG spike 都必须另开计划和资源，不能影响 tracker、attention、EventEngine、gaze DDS、Botified 输出或 PC gate。候选细节保留在 `docs/product-candidate-visual-memory-rag.md`，本计划不拆 memory 开发任务。

## 10. 风险与控制

| 风险 | 控制 |
| --- | --- |
| 规则复杂度失控 | 只增强现有事件；不加大量新事件；阈值集中在 `EventConfig` |
| 短遮挡误合并不同人 | alias 窗口短、条件保守、只用于去重，不当长期身份 |
| waving 误报 | evidence gate 更严格；宁可少触发，不刷屏 |
| val-data 过拟合 | 先锁 forbidden events 和负例，再调正例触发时间 |
| scene context 变成新事件风暴 | 默认只在 `visual_state` 和可视化中出现，不进 Botified |
| Visual Memory 拖入主链路 | v0.2 主线不接 memory；spike 必须显式 opt-in、可关闭 |

## 11. 开发原则

这些原则必须写进实现 review：

- KISS：一个实时主链路，一个事件引擎，一个 scene context 输出，一个高频 DDS gaze，一个低频 Botified 出口。
- DRY：事件规则只在 server 实现一次；CLI 不复刻规则；测试工具不成为第二套业务逻辑。
- YAGNI：没有验收需求前，不加模型训练、人脸/ReID、数据库、后台、复杂报告、长期治理流程。
- TDD 聚焦核心 runtime：事件规则、协议兼容、DDS/Botified 边界、PC E2E。
- 不测试测试工具；测试工具只保留能保护 runtime 的轻量 smoke。
- 代码优先清晰，只有真实重复和复杂度出现时才抽象。

## 12. 团队 review 结论

产品视角结论：

- v0.2 应优先解决商店门口揽客体验中的重复通知、误触发、目标切换和不可解释问题。
- passing by 和 approaching 仍然合理，因为这是门口揽客的核心触发场景，但必须由时间窗口、机器人/相机运动 gating 和 cooldown 管住。
- scene context 应先作为状态提供给 agent/调试，不应急着变成新的 Botified 事件。
- Visual Memory 有长期价值，但不是下一版主线。

研发视角结论：

- 主改动集中在 `EventEngine`，processor/CLI 保持薄边界。
- 现有 tracker 输出已经足够，不需要 ReID 或第二模型。
- evidence 和 lifecycle 可以增量加到现有 proposal/emit 流程，不需要重写架构。
- `val-data` 和 PC local E2E 是 v0.2 的通过依据；真机和 RK3588 是后续验证，不阻塞本版本。

本计划已经按 KISS/DRY/YAGNI 收敛，可以交付开发团队执行。
